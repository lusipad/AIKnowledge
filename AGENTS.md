# OKK 仓库执行约定

## 响应语言

- 默认使用中文简体回复。

## 执行方式

- 对于实现类任务，默认持续工作直到任务真正完成，不要做阶段性汇报。
- 除非满足以下条件，否则不要中途停下来总结进度：
  - 用户明确要求汇报
  - 出现真实阻塞，无法继续
  - 需要确认危险操作
- 工具调用前的说明压缩到最低；如非必要，不要为了“同步进度”而额外输出。
- 用户要求“持续执行直到全部完成”时，该要求高于默认协作式汇报风格。
- 完成一个 change 后，直接继续下一个 change，不要做过程性播报。
- 每个 change 都按照 计划 - 执行 - 测试 完整执行
- 完成一个 change 后，git commit 和 push。
- 当用户说“继续开发”“继续做”“别停直到全部完成”等类似指令时，必须先识别并维护一个显式 backlog，再开始执行。
- backlog 默认来源包括：`README.md`、`docs/`、测试、代码中的占位实现、`TODO/FIXME`、以及已声明但未完全实现的接口或能力。
- 只要 backlog 非空，就禁止结束当前轮次；完成一个 change 后必须自动进入 backlog 中下一项。

## Backlog 规则

- 开始实现前，先整理剩余未完成项，并按优先级排序。
- 优先级固定为：
  - 文档、README、API 说明中已承诺但未真正完成的能力
  - 明确的占位实现或“后续替换”实现
  - 生产可用性缺口：可靠性、异步化、权限、隔离、数据治理
  - 可观测性、运维、评估与治理能力
  - 纯体验优化或低风险重构
- 除非用户明确说“只做到 MVP 范围”，否则不能把“MVP 已闭环”解释为“全部完成”。
- `README`、`docs`、注释中写明的限制项、占位项、后续项，默认都属于 backlog，直到被真正消除或被用户明确降级。

## 完成定义

- 不要把“已经可用”、“已经可交付”、“主链路已打通”默认视为任务完成。
- 只有在以下条件同时成立时，才能停止并给出最终结论：
  - 用户请求的目标已经实现
  - 相关代码、脚本、文档、测试已经同步更新
  - 已完成必要验证，且没有明显残留的高优先级缺口
  - 当前继续推进不会再带来明确的质量提升或交付价值
  - backlog 已清零，或 backlog 中剩余项都已被用户明确排除在本轮范围外
- 如果仍存在明显的下一步高价值工作，应继续执行，而不是先停下来“汇报已完成”。
- 如果任务表述为“完整做完”“端到端交付”“直接按最优方案做”，则默认需要继续推进到生产可用边界，而不是只完成 MVP 主流程。

## 禁止过早收敛

- 不要因为通过了一轮测试、完成了一次提交、或者已经推送远端就停止。
- 不要把“我认为这已经够了”作为停止依据，停止依据只能来自用户目标、真实阻塞、危险操作确认点，或确实没有剩余高价值工作。
- 当你判断“已经完成”时，必须先反向检查：
  - backlog 是否已经清零
  - 是否还有 README / docs / API / tests 中明确承诺但未完成的能力
  - 是否还有代码中的占位实现、后续替换点、或临时兜底实现
  - 是否还有未覆盖的关键链路
  - 是否还有明显缺失的工程化项
  - 是否还有可以直接继续完成且无需用户确认的工作
- 如果上述任一答案为“有”，则继续工作，不要结束。

## Git 与危险操作

- 涉及 `git reset --hard`、删除文件/目录、批量不可逆修改时，必须先明确确认。
