## 1. MVP 目标

MVP 的目标不是一次性构建完整平台，而是验证这套系统在单团队、单仓库场景中是否能稳定带来价值。

MVP 需要回答三个问题：

1. 能否低侵入地采集有用上下文。
2. 能否稳定召回与当前任务相关的规则和案例。
3. 能否让开发者明显减少重复背景输入。

## 2. MVP 范围

## 2.1 必做范围

- 单团队、单仓库接入。
- IDE/CLI 至少一条接入链路。
- 会话与关键事件采集。
- 仓库级和路径级规则配置。
- 基础知识抽取与审核。
- Hybrid Retrieval。
- 来源追踪。
- 基础反馈收集。
- `tenant/team` 请求级隔离与资源 ACL。
- PostgreSQL + Alembic + 原生 `pgvector` 检索链路。
- 外部 IAM Bearer JWT 与组织/团队作用域同步。
- 跨系统 SCIM / 目录自动回写。
- 组织级跨仓知识图谱最小闭环。

## 2.2 不做范围

- 多租户复杂治理。
- 组织级全局图谱。
- 自动复杂失效系统。
- 全自动知识发布。
- 自动复杂失效系统。

## 3. MVP 成功标准

### 3.1 产品成功标准

- 试点开发者能明显感受到背景重复输入减少。
- 关键仓库规则能够稳定被检索到。
- 至少一种案例知识能在第二次相似任务中复用成功。

### 3.2 技术成功标准

- 检索 `P95 < 1.5s`
- Top5 相关性达到试点可用标准
- 审核发布链路可稳定运行
- 全链路来源可追踪
- 已具备可重复执行的检索 benchmark / 压测入口，可直接验证 `P95` 目标

## 4. 试点范围建议

推荐选择以下类型仓库：

- 业务边界清晰
- 规则较稳定
- 团队成员有 AI 使用习惯
- 有一定历史案例沉淀
- 不是超大单体，也不是过于琐碎的小仓库

不建议首批试点选择：

- 高频变动且无明确规则的探索性项目
- 高敏感强隔离项目
- 缺乏领域 owner 的混乱仓库

## 5. MVP 交付范围拆解

## 5.1 客户端能力

- 创建会话
- 上报关键事件
- 拉取仓库/路径配置
- 调用检索接口
- 提交反馈

## 5.2 服务端能力

- Session 管理
- Event Ingest
- 基础 Signal 识别
- 知识抽取任务
- 审核管理
- Retrieval 服务
- Config 服务
- 审计与基础日志

## 5.3 存储能力

- PostgreSQL
- 向量索引
- 搜索索引
- 对象存储

## 6. 实施阶段

## 6.1 阶段一：接入与规则供给

目标：先把近场上下文和规则供给跑通。

交付：

- 客户端接入
- 会话创建与事件上报
- 仓库级 / 路径级配置下发
- 基础检索接口

验收：

- AI 能稳定获得仓库规则和路径规则

## 6.2 阶段二：知识沉淀

目标：让高价值会话可以进入知识库。

交付：

- Signal 识别
- 抽取任务
- 规则/案例/流程三类知识结构
- 审核后台基础能力

验收：

- 至少一类知识可完成“会话 -> 抽取 -> 审核 -> 发布”闭环

## 6.3 阶段三：检索增强

目标：提升召回质量和上下文相关性。

交付：

- BM25 + Dense Retrieval
- Metadata Filter
- 基础 rerank
- Context Pack 生成

验收：

- 试点评审中 Top5 相关性达标

## 6.4 阶段四：反馈闭环

目标：让系统知道哪些知识真正有用。

交付：

- 采纳反馈
- 忽略/过时反馈
- 基础评分回流
- 检索日志与使用日志

验收：

- 能通过反馈调整知识质量分和排序结果

## 7. 角色分工建议

### 7.1 产品

- 明确试点场景
- 定义验收标准
- 组织用户访谈与反馈收集

### 7.2 平台后端

- 实现会话、知识、检索、配置、审核接口
- 搭建存储与消息链路

### 7.3 AI 工程

- 设计信号规则
- 设计抽取模板
- 设计 rerank 与上下文压缩策略

### 7.4 客户端 / IDE

- 完成接入层
- 上报上下文
- 集成检索结果注入

### 7.5 试点团队 Owner

- 提供规则源
- 审核知识
- 参与效果验证

## 8. 里程碑计划建议

## 8.1 第 1 周

- 明确试点仓库与用户
- 冻结 MVP 范围
- 确定数据边界与安全要求

## 8.2 第 2-3 周

- 完成会话采集与配置下发
- 完成基础检索链路

## 8.3 第 4-5 周

- 完成知识抽取与审核链路
- 建立规则/案例/流程三类知识模型

## 8.4 第 6 周

- 完成反馈闭环和看板基础数据
- 进行试点验收

## 9. 核心验收用例

### 用例 1：新需求实现

- 输入：新需求描述 + 当前目标文件
- 预期：返回仓库规则、模块约束、历史相关实现

### 用例 2：典型错误修复

- 输入：错误日志 + 当前 diff
- 预期：返回相似案例、修复建议、回归检查项

### 用例 3：路径级规范

- 输入：编辑指定目录下文件
- 预期：自动命中路径级规则

### 用例 4：知识反馈

- 输入：开发者标记某条知识“过时”
- 预期：系统记录反馈并影响后续排序

## 10. 风险与兜底策略

### 10.1 抽取效果不稳定

兜底：

- 初期优先人工审核
- 知识条目草稿态默认不参与检索

### 10.2 检索噪声过高

兜底：

- 缩小作用域
- 降低默认召回量
- 强化路径过滤

### 10.3 客户端接入成本偏高

兜底：

- 先做 CLI / MCP 简版接入
- 减少首期采集字段

### 10.4 用户无感

兜底：

- 聚焦两个高频场景：新需求、bug 修复
- 优先提升最常见任务的命中率

## 11. MVP 结束后的决策门槛

若满足以下条件，建议进入下一阶段：

- 试点团队愿意持续使用
- 关键规则命中率稳定
- 至少一类案例被复用多次
- 用户主观反馈明显为正向

若不满足，建议回到以下方向优化：

- 收窄范围
- 优化检索链路
- 优化知识抽取模板
- 提高配置质量
