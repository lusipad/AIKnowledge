- 文档版本：`v1.0`

- 关联文档：`docs/Proposal.md`、`docs/Design-Arch.md`

## 1. 产品概述

本产品面向 AI Coding 场景，目标是建设一套团队级知识与记忆系统，让 AI 在编码、排障、改造、需求开发过程中，能够自动获得团队规则、业务背景、历史案例和操作流程，从而提高输出质量、稳定性与复用效率。

该产品不是传统文档检索系统，也不是简单的聊天记录存档系统，而是一个围绕“上下文供给”构建的研发基础设施产品。

## 2. 产品目标

### 2.1 业务目标

- 降低开发者重复向 AI 解释背景的成本。
- 提高 AI 在真实项目中的首轮建议可用率。
- 让团队隐性经验沉淀为可持续复用的组织资产。
- 为后续代码智能体、多 Agent、自动修复等能力提供底座。

### 2.2 产品目标

- 自动采集高价值上下文与开发活动信号。
- 自动或半自动沉淀为结构化知识。
- 在任务执行时按需检索并生成上下文包。
- 对知识进行审核、版本、权限、失效和反馈管理。

## 3. 用户与角色

### 3.1 开发者

诉求：

- 不想反复解释项目背景。
- 希望 AI 更懂当前仓库和团队习惯。
- 希望快速找到相似修复案例和团队规范。

### 3.2 技术负责人 / TL

诉求：

- 希望把团队规则、最佳实践和关键经验统一沉淀。
- 希望减少低质量代码建议和重复踩坑。
- 希望对知识质量和使用效果有可见性。

### 3.3 平台管理员

诉求：

- 希望有统一配置、权限和审计机制。
- 希望控制知识作用域、敏感数据和模型行为。

### 3.4 审核者 / 知识运营者

诉求：

- 希望审核重要知识候选，维护高质量规则库。
- 希望下线过期和误导性知识。

## 4. 核心问题陈述

当前 AI Coding 工具普遍存在以下问题：

- 上下文依赖人工搬运。
- 对当前项目的真实约束感知不足。
- 历史会话和案例无法稳定复用。
- 对团队级规则缺少统一供给能力。
- 没有清晰的来源追踪和效果评估。

因此，产品必须同时解决四个问题：

1. 怎么采集真正有价值的上下文。
2. 怎么把这些上下文沉淀成可管理的知识。
3. 怎么在正确时机给 AI 正确的内容。
4. 怎么持续判断这些知识是否真的有效。

## 5. 使用场景

### 5.1 新需求开发

用户发起新任务时，系统自动提供：

- 当前业务域背景。
- 相关模块说明。
- 历史实现约束。
- 当前仓库规则。
- 可参考案例。

### 5.2 缺陷修复

用户打开报错文件或粘贴错误日志时，系统自动提供：

- 相似错误案例。
- 历史修复路径。
- 相关测试清单。
- 可能影响模块。

### 5.3 代码改造 / 重构

当用户准备大范围修改时，系统自动提供：

- 相关目录约定。
- 历史改造经验。
- 常见风险与回归项。
- 依赖边界说明。

### 5.4 新成员上手

系统按仓库和路径注入：

- 团队规则。
- 核心目录说明。
- 常见坑点。
- 发布与回归流程。

## 6. 产品范围

## 6.1 本期范围

- AI 会话与关键事件采集。
- 规则、案例、流程三类知识沉淀。
- 基于仓库 / 路径 / 团队的知识作用域管理。
- 混合检索与上下文包生成。
- 审核、反馈、版本与失效管理基础能力。

## 6.2 非本期范围

- 企业全域知识统一平台。
- 重型跨组织知识图谱。
- 全量历史对话精细化解析。
- 自动化复杂工作流编排平台。

## 7. 核心功能需求

## 7.1 事件采集

### 7.1.1 功能描述

系统采集 AI Coding 过程中的关键事件，用于后续信号识别和上下文构建。

### 7.1.2 采集内容

- Prompt 摘要。
- 模型响应摘要。
- 工具调用记录。
- 命令执行结果摘要。
- 当前文件与变更文件路径。
- 测试结果摘要。
- 当前任务元数据。

### 7.1.3 验收标准

- 能按 `session_id` 聚合事件。
- 客户端感知延迟低。
- 支持失败重试和异步上报。

## 7.2 信号识别

### 7.2.1 功能描述

从采集到的事件中识别高价值知识候选。

### 7.2.2 信号类型

- 修复成功。
- 规则形成。
- 案例可复用。
- 流程性操作明确。
- 反复出现的问题被稳定解决。

### 7.2.3 验收标准

- 能为候选输出 `signal_type`、`confidence`、`priority`。
- 能过滤低价值噪声会话。

## 7.3 知识抽取

### 7.3.1 功能描述

将高价值会话候选提取为结构化知识单元。

### 7.3.2 支持类型

- 规则类知识。
- 案例类知识。
- 流程类知识。

### 7.3.3 结构模板

- 标题。
- 背景。
- 结论。
- 适用范围。
- 来源。
- 置信度。
- 标签。
- 失效条件。

### 7.3.4 验收标准

- 能从候选生成结构化知识对象。
- 能保留来源引用。
- 能对重复内容进行合并提示。

## 7.4 知识治理

### 7.4.1 功能描述

支持知识审核、发布、下线、版本管理和降权。

### 7.4.2 核心能力

- 审核通过 / 驳回。
- 版本更新。
- 生效 / 失效时间。
- 质量评分。
- 采纳反馈联动。

### 7.4.3 验收标准

- 审核链路闭环可用。
- 能查看知识历史版本。
- 能标记失效并停止被检索。

## 7.5 检索与上下文包生成

### 7.5.1 功能描述

根据任务场景检索相关知识，并生成可注入 AI 的上下文包。

### 7.5.2 输入维度

- 任务描述。
- 当前仓库 / 分支。
- 当前文件 / 路径。
- 当前 diff。
- 用户身份与权限。

### 7.5.3 输出内容

- 规则列表。
- 相关案例。
- 流程建议。
- 来源引用。
- 总结摘要。

### 7.5.4 验收标准

- 能按查询类型进行路由。
- 能控制 token 预算。
- 能返回来源与作用域。

## 7.6 配置中心

### 7.6.1 功能描述

统一管理团队级与仓库级 AI 行为配置。

### 7.6.2 配置内容

- 提示词模板。
- 仓库规则。
- 路径规则。
- 检索策略。
- 抽取模板版本。
- 审核策略。

### 7.6.3 验收标准

- 支持按范围下发配置。
- 支持版本控制与回滚。

## 7.7 反馈与评估

### 7.7.1 功能描述

采集用户对知识和检索结果的反馈，用于优化排序、发布和失效策略。

### 7.7.2 反馈类型

- 采纳。
- 忽略。
- 错误。
- 过时。
- 有帮助但不完整。

### 7.7.3 验收标准

- 能记录知识采纳结果。
- 能将反馈回流到评分体系。

## 8. 非功能需求

### 8.1 性能

- 检索响应 `P95 < 1.5s`
- 配置拉取 `P95 < 300ms`
- 采集对客户端影响低感知

### 8.2 可用性

- 核心检索可用性 `99.9%`
- 采集与配置服务可用性 `99.9%+`

### 8.3 安全性

- 目标架构支持多租户隔离；当前试点版仅透传租户/团队上下文，不提供隔离能力。
- 支持敏感信息脱敏。
- 支持检索与注入日志审计。

### 8.4 可扩展性

- 支持新增知识源。
- 支持新增 Agent / IDE 客户端。
- 支持新增检索后端与重排策略。

## 9. 交互流程

## 9.1 开发者任务开始

1. 用户在 IDE / CLI 中发起任务。
2. 客户端收集当前任务与工作区上下文。
3. 系统执行检索与路由。
4. 返回上下文包给 AI。
5. AI 基于上下文执行任务。

## 9.2 高价值会话沉淀

1. 系统采集关键事件。
2. 信号引擎判断是否值得沉淀。
3. 触发知识抽取。
4. 进入审核 / 自动发布。
5. 建立检索索引。

## 10. 优先级划分

## 10.1 P0

- 事件采集
- 仓库 / 路径规则下发
- 基础检索
- 来源追踪

## 10.2 P1

- 知识抽取
- 审核与发布
- 反馈回流
- 基础失效管理

## 10.3 P2

- Prompt Cache
- 短期上下文压缩
- 高级 rerank
- 自动降权

## 11. 成功指标

### 11.1 北极星指标

- AI Coding 任务中“重复背景输入次数”显著下降。

### 11.2 过程指标

- 高价值会话沉淀率。
- 检索相关性。
- 规则命中率。
- 案例复用率。
- 反馈采纳率。

### 11.3 结果指标

- 首轮建议采纳率提升。
- 任务完成时间下降。
- 返工和回滚率下降。

## 12. 风险与依赖

### 12.1 风险

- 沉淀噪声过高。
- 检索噪声污染上下文。
- 过期知识误召回。
- 客户端采集边界不清引发隐私风险。

### 12.2 外部依赖

- IDE / Agent 接入能力。
- 认证与权限系统。
- 模型服务。
- 检索与存储基础设施。

## 13. 里程碑建议

### `M1`

- 完成单仓采集、规则配置、基础检索。

### `M2`

- 完成知识抽取、审核、发布。

### `M3`

- 完成反馈闭环与效果看板。

### `M4`

- 完成多项目推广与策略优化。

## 14. 发布标准

满足以下条件可进入试点：

- 单仓规则召回稳定。
- 至少一种知识类型完成从采集到使用闭环。
- 能够在审计日志中看到完整来源链路。

满足以下条件可扩大范围：

- 检索相关性达到目标阈值。
- 用户采纳率达到试点要求。
- 权限与敏感信息问题可控。
